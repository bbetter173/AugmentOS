# iOS Jetsam Stress Test

End-to-end harness for measuring how many local miniapp WebViews the
MentraOS app can hold mounted before iOS jetsam starts killing things.

## What this test answers

1. **N_max foreground** — how many simultaneously mounted dummy WebViews
   can the app hold while the user is looking at it.
2. **N_max background** — same but with the app backgrounded (the realistic
   production scenario, where iOS jetsam is much more aggressive).
3. **Time-to-jetsam (host app)** — with N WebViews mounted and BLE active,
   how long does the host app survive in background before iOS terminates it.
4. **Memory growth shape** — does each WebView add roughly constant
   overhead, or do they grow over time (leak).
5. **BLE survivability under pressure** — does jetsam'ing WebViews knock
   the BLE session down too.

The product requirement is that **miniapps stay running while the phone
screen is off and the app is in background, with glasses connected**.
This test exists to characterize whether the persistent-WebViews strategy
can satisfy that, not to choose between strategies.

## Pre-requisites

- A real iOS device paired with Xcode. Simulator does not implement real
  jetsam — its WebContentProcess kills are best-effort and not comparable.
  Default device is `Israelov` (iPhone 15); pass another name as 2nd arg.
- App built and installed on the device (any debug or release build with
  this branch's instrumentation in it).
- Maestro CLI installed (`brew install maestro`).
- Super Mode enabled in the app (Konami code: swipe ↑ ↓ ↑ ↓ ← ← anywhere,
  then flip the toggle).
- Glasses paired (so the BLE session is active during the test). Optional
  but realistic.

## What the test does

1. Streams the iOS unified log to a file, filtering for our process,
   `STRESS:` lines, jetsam, memorystatus, and WebKit memory pressure.
2. Launches the app via `xcrun devicectl`.
3. Replays a Maestro flow that navigates to the stress-test screen, picks
   a per-app heap size, hits "Start logging", and mounts N dummy WebViews.
4. Dwells for the scenario-specific duration, polling app liveness via
   `xcrun devicectl device info processes` every 30s.
5. Stops the log stream, parses STRESS: lines into a CSV, and prints a
   summary.

## Running

```bash
cd mobile

# Foreground scenario (default): mount 25 dummies @ 25 MB each, watch 2 min
./scripts/stress-test/run-jetsam-test.sh foreground

# Background: mount 5 @ 25 MB, send to home, watch 10 min
./scripts/stress-test/run-jetsam-test.sh background

# Long background: mount 5 @ 25 MB, send to home, watch 60 min
./scripts/stress-test/run-jetsam-test.sh long

# Leak hunt: mount 5 @ 25 MB, idle foreground, watch 60 min
./scripts/stress-test/run-jetsam-test.sh leak

# Override the per-app heap and mount count
MB_PER_APP=100 MAX_MOUNTS=5 ./scripts/stress-test/run-jetsam-test.sh background
```

Output lands in `scripts/stress-test/runs/<timestamp>-<scenario>/`:

- `raw.log` — filtered unified log (STRESS:, jetsam, memorystatus, app process)
- `events.csv` — parsed STRESS: samples and events
- `meta.json` — run metadata (device, scenario, parameters)
- `maestro.log` — Maestro test runner output
- `launch.log` — devicectl launch output

## How instrumentation works

Three native/JS additions feed the test:

1. **`MiniappHost.tsx` `handleTerminate`** records every WebView Web
   Content Process kill into `useStressTestStore` and emits
   `STRESS: event {...}` to console.
2. **`MemoryWarningMonitor.tsx`** listens for RN `AppState` "memoryWarning"
   events (which iOS emits via `applicationDidReceiveMemoryWarning`) and
   records them.
3. **`CoreModule.getMemoryMB()`** is a new Expo Function exposed from
   `mobile/modules/bluetooth-sdk/ios/BluetoothSdkModule.swift` that wraps
   the existing `MemoryMonitor.currentMemoryMB()` (reads RSS via
   `task_info`). The stress-test screen polls it once a second.

The test screen itself is at `src/app/miniapps/settings/stress-test.tsx`
and the dummy WebView HTML is generated by
`src/utils/stressTest/dummyHtml.ts` — it allocates `mb` MB of `Uint8Array`
and touches it every 5s so iOS doesn't lazily reclaim it.

## What "STRESS:" lines look like

Once "Start logging" is tapped, the screen emits one structured log line
per second:

```
STRESS: sample {"at":1715291130000,"residentMB":412.3,"mounted":7,"terminated":2,"memwarn":1}
```

And one line per event:

```
STRESS: event {"packageName":"com.mentra.stress.dummy.3","at":1715291131500,"kind":"terminate","residentMB":405.1,"memWarnCount":1}
STRESS: mount {"pkg":"com.mentra.stress.dummy.4","mb":25,"at":1715291132000}
```

The CSV produced after the run has one row per STRESS line.

## Reading the results

The two key signals to look for:

- **Terminate events** in the CSV with `kind=terminate` are individual
  WebView kills. Time-correlate with the `mounted` and `residentMB`
  columns to find when iOS started reaping.
- **App death** appears in the harness's polling output as the first
  iteration where `alive=0`. The CSV stops at that point. The raw log
  may have a `kernel: jetsam` line just before — search the raw.log for
  `jetsam` and look at the timestamp.

If the host app survives the entire dwell window, iOS is happy with the
configuration and you can step up the load (more dummies, bigger heap,
or longer background dwell) on the next run.

## Caveats

- iOS-only. Android jetsam is a totally different story and `getMemoryMB`
  returns 0 there.
- `log stream --device` may need `sudo` on some macOS versions; if you get
  a permission error, run `sudo -v` once before the script.
- Maestro can't actually lock the device screen, only background the app
  via Home. To test "screen off" precisely, lock the phone manually after
  the Maestro flow finishes mounting (it will print a notice).
- The Web Content Process for each WebView is a separate OS process, so
  jetsam can kill individual WebViews without taking down the host app.
  Only when host RAM pressure is high does the host app itself get reaped.
