# Spike: Org Creation Fix Not Deployed to Production

## Overview

**What this doc covers:** Investigation into why user `james@groupe.nyc` is still stuck in an infinite org creation loop on production (`console.mentra.glass`) despite the comprehensive fix landed on `dev` via PR #2108 (`cloud/org-bug`).

**Why this doc exists:** James reported the issue on Feb 27, was told a fix would ship within a week, checked back Mar 3 (still broken), and again yesterday (still broken). The fix exists — it was merged to `dev` on ~Feb 28 — but was **never merged to `main` or `staging`**, so production is still running the original broken code.

**Who should read this:** Anyone with merge access to `main`. This is a deployment gap, not a code gap.

## Background

Issue [043-org-creation-bug](../043-org-creation-bug/) fully diagnosed and fixed the org creation infinite loop. The fix landed in eight files across backend and frontend, addressing:

- Slug collision (`generateSlug` has no uniqueness handling)
- `createPersonalOrg` not idempotent (no check for existing org, no E11000 catch)
- Frontend `useEffect` infinite retry loop (unstable deps in dependency array)
- `user.save()` VersionError under concurrency

The fix was merged to `dev` via PR #2108 (commit `8325d66be`). Two follow-up commits refined it:

- `0509ad382` — deterministic slug suffix closes TOCTOU race
- `4bb58318a` — idempotency check scoped to admin role + surface bootstrap failures

### 2. Production is running the original broken code

Checked `origin/main:cloud/packages/cloud/src/services/core/organization.service.ts`:

```typescript
// main branch — NO idempotency check, NO E11000 handling, NO slug uniqueness
public static async createPersonalOrg(user: UserI): Promise<Types.ObjectId> {
  const personalOrgName = `${user.profile?.company || user.email.split("@")[0]}'s Org`;
  const slug = await generateSlug(personalOrgName);
  const org = new Organization({ name: personalOrgName, slug, ... });
  await org.save();  // Throws E11000 if slug exists → 500 → frontend retries forever
  return org._id;
}
```

And `origin/main:cloud/websites/console/src/pages/OrganizationSettings.tsx`:

```typescript
// main branch — infinite retry loop still present
useEffect(() => {
  const createPersonalOrg = async () => {
    if (!currentOrg && !orgLoading && !isCreatingOrg) { ... }
  };
  createPersonalOrg();
}, [currentOrg, orgLoading, ensurePersonalOrg, isCreatingOrg]);
//                          ^^^^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^
//                          new ref every render  toggles on fail → retrigger
```

And `origin/main:cloud/websites/console/src/context/OrganizationContext.tsx`:

```typescript

// AND it calls api.orgs.create() directly (frontend-initiated org creation)
const ensurePersonalOrg = async (): Promise<void> => {
  // ...
  const newOrg = await api.orgs.create(personalOrgName);  // POST /api/orgs → 500
  // ...

```

### 3. James's error pattern in BetterStack confirms the exact 043 bug

Queried BetterStack for `james@groupe.nyc` errors in the last 7 days:

| Timestamp | Error | Slug |
|---|---|---|
| 2026-03-04 22:05:13–22:06:17 | `E11000 duplicate key error ... slug_1` | `james-s-organization` |
| 2026-03-04 22:05:13–22:06:16 | `E11000 duplicate key error ... slug_1` | `james-s-org` |
| 2026-03-03 18:39:48–18:40:36 | Same | Same |
| 2026-02-27 (original report) | Same | Same |

**Both slug variants** collide — `james-s-org` (generated by `createPersonalOrg` on the backend via `getConsoleAccount`) and `james-s-organization` (generated by the frontend's `ensurePersonalOrg` via `POST /api/orgs`). Another user named "james" already owns both slugs. Neither code path handles the collision. The frontend retries in a tight loop (~200ms between attempts).

The pattern is:

```
GET  /api/orgs           → 200 (0 orgs)
GET  /api/console/account → 500 (E11000: slug "james-s-org" exists)
POST /api/orgs            → 500 (E11000: slug "james-s-organization" exists)
GET  /api/orgs           → 200 (0 orgs — still no org)
POST /api/orgs            → 500 (loop continues)
```

This is exactly the cascade documented in the [043 spike](../043-org-creation-bug/spike.md), sections 1–5.

### 4. Other users are still hitting this too

BetterStack shows org creation E11000 errors for other users in the last 48 hours:

- `contact-s-organization` — rapid-fire 500s on Mar 5 at 19:47–19:48 (~25 errors in 2 minutes)
- `david-s-organization` — Mar 5 at 19:55 and 20:07

These are all different users whose email prefix collides with an existing org slug. Every new user whose name matches an existing slug will hit this until the fix is deployed.

### 5. The console frontend deployment is also stale

The console (`console.mentra.glass`) appears to be a static site built from the same branch as the cloud backend. The production console still has:

- `ensurePersonalOrg` as a plain arrow function (not `useCallback`) — new reference every render
- `isCreatingOrg` in the `useEffect` dependency array — toggles on fail, re-triggers the effect
- Frontend-initiated `api.orgs.create()` call — the fixed version removes this entirely and relies on backend bootstrap

## Conclusions

| Problem | Severity | Fix status |
|---|---|---|
| `cloud/org-bug` fix not on `main` | **Critical** — blocks every new user whose name collides with an existing slug | Code done, deployment missing |
| Frontend infinite retry loop on prod | **Critical** — generates thousands of 500s per affected user session | Code done, deployment missing |
| Multiple users actively blocked | **High** — james@groupe.nyc (since Feb 27), plus at least 2 others in last 48 hours | Workaround available (fresh account + invite) |

**This is not a code problem. The fix is complete and reviewed.** The only action needed is merging `dev` → `staging` → `main` (or cherry-picking the org-bug commits to `main`).
